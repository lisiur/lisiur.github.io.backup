使用FP进行空值检查和错误处理的实践

Posted on 2017-03-26 | In fp | Comments: | Views:

如果说Java百分之八十的错误是空指针异常的话，那么我说Javascript百分之六十的错误是“Cannot read property ‘**’ of undefined”大家应该不会有意见吧（至少目前对于我这个小实习生来说是这样的= =）

很多时候，我们都会默认拿到的数据是按照约定的结构返回的，如果数据真的都是这样的话，大概是可以按照我们预期的想法执行。但很多时候我们拿到的数据会跟你耍些小把戏，比如藏起自己的一只手，如果你本来是要给这个数据带上手套，但当你发现他连手都没有的话就会很尴尬了。你可以说这是他的锅，一个人怎么能没有手呢。然而意外总会发生，不是么？

举个简单的例子，我们有个对象obj, 和颜色表colorTB定义如下：

我们的任务是根据obj的config中的color获取其在colorTB中对应的color值。（该例子中， 因为obj.config.color => ‘red’, 所以其对应的color值是colorTB.red => ‘#FF0000’）

我们目前唯一能保证的是，会有obj这个对象和colorTB对象，其他都是未知的。即，有可能obj的config中没有color属性；或obj根本没有config属性；也或obj的color在colorTB中没有对应值。

这种方式固然可以处理这种情况，但这种处理方式在我看来有如下几个问题：

  1. 处理过程过于繁琐，需要编码者掌控所有细节问题
  2. 错误处理过于隐蔽，错误细节没有暴露出来（即如果不进入函数内，不知道是因为obj没有color而出错，还是因为colorTB没有对应值而出错。不要告诉我你想通过抛出不同错误来区分，我想你大概也是不想每次执行这个操作都要用try-catch吧）
  3. 不易扩展（和第一点有关，因为细节问题过多）
  4. 不易读（和第一点有关）

其实这个问题，归根结底就是我们一开始提出的“Cannot read property ‘**’ of undefined”问题，无论是obj拿不到color还是colorTB拿不到给定的key对应的value。对于vue用户来说，一种更常见的场景是，组件中传的prop中如果有一个具有至少两层深度（Object中含有Object）的Object，我们在做组件时默认是认为该Object每个key（包括后代key）都是有value的，但事实上很有可能是我们传的Object并不总是满足这个要求。对于这种情况有两种解决方案，一种是在组件开发者在组件上做空检测，一种是组件使用者在传值前做空检测。我个人觉得在组件上做空检测是比较好的方式，但现实是，作为组件使用者，大多数组件并不是我们所能控制的，且很多组件并没有做空检测（或许是考虑到组件的纯洁性）。因此我们不得不屈服在其淫威下自己来做空检测了。

做空检测是必要的，虽然在大多数情况下获取对象的深度属性是没有问题的，只有在获取那些自己不可控的对象的深度属性值时才需要做空检测，但如果我们每次获取一个这样的对象的某个深度属性值时，都要用到if-else的话，我觉得我会放弃程序员这个职业的。人生苦短，何必这么为难自己 :)

还好在我坚定放弃之前，在前人的基础上摸索到了一套方案，没错，就是FP

这儿，我们一共创建了三个类，Either，Left和Right，其中Left和Right是Either的子类，实际上我们并不会创建Either的实例，他在这只是为了概念和继承链上的完整性。Right类和Container基本没有什么区别，我们需要关注下Left类，他最有意思的是map方法返回的永远是其自身，也就是说任何加在他身上的操作都会被无视掉。这一点很重要！还有一个需要关注的是，Either类提供了一个处理Either类型的方法either。
