关于事件依赖的思考和实践

发表于 2017-01-24   |  

在使用vue做个人中心的绑定手机功能时，遇到这样的需求：

  • 有密码输入框，手机号输入框，验证码输入框，验证码发送按钮和绑定按钮
  • 当密码输入框失去焦点，要发送一个异步请求校验密码是否正确
  • 当手机号输入框失去焦点，要本地校验手机号格式是否正确
  • 只有当密码校验正确 && 手机号校验正确 验证码发送按钮才能生效
  • 只有当密码校验正确 && 手机号校验正确 && 验证码输入框已输入内容 绑定按钮才能生效

为了能够聚焦本次讨论的焦点，我暂时把需求简化一下：

  • 有密码输入框，手机号输入框和确定按钮
  • 当密码输入框失去焦点，要发送一个异步请求校验密码是否正确
  • 当手机号输入框失去焦点，要本地校验手机号格式是否正确
  • 只有当密码校验正确 && 手机号校验正确 确定按钮才能生效

这种方案看似能满足需求，但有个问题：如果某一时刻用户既触发了checkPass事件又触发了submit事件，在下面两种情况下将导致与预期不符的事件发生（假设phoneNumberValid始终为true）：

  1. 若passwordValid本来的值为false，此时checkPass函数需要发送一个异步请求才能确定password是否正确，如果异步请求判断正确，将会在异步请求结束才修改passwordValid值为true，而在这之前submit函数中计算的canSubmit为false，导致不能发送确认请求。
  2. 若passwordValid本来的值为true，此时checkPass函数需要发送一个异步请求才能确定password是否正确，如果异步请求判断不正确，将会在异步请求结束才修改passwordValid值为false，而在这之前submit函数中计算的canSubmit为true，导致发送了本不该发送的确认请求。

仔细分析会发现发生上面的情况，原因有二：

  1. submit事件依赖checkPhone事件和checkPass事件对共享数据状态的影响
  2. submit事件依赖的checkPhone事件含有异步请求

若submit事件不依赖checkPhone事件和checkPass事件对共享数据状态的影响，那么submit事件可以独立执行不会出现上面的情况；
若submit事件依赖的checkPhone事件不含有异步请求，那么所有事件会按照用户的操作顺序执行，也不会出现上面的情况；

所以问题的关键就是submit在执行时要确保其依赖的事件要执行完毕（包含所有异步请求）。对于不含有异步请求的事件，其函数执行完毕，该事件也就执行完毕了可以不用对其进行特殊处理；但对于含有异步请求的事件，当其函数执行完毕还要等异步请求执行完毕才能算该事件结束，此时所有对该异步请求的结果有依赖的事件都需要在该异步请求结束的回调里执行。

这种方案中添加了hasSubmit变量，该变量记录是否含有submit事件需要执行。该方案的核心思想是：把submit执行时机交给其依赖的事件处理。hasSubmit的初始值为-1:

  1. 只执行submit，可正常执行
  2. 只执行checkPass，可正常执行checkPass而不触发submit事件
  3. 在checkPass执行异步请求的过程中执行submit，则submit会被挂起直到checkPass的异步请求结束才会执行submt

这里面的hasSubmit就类似一个锁机制，在checkPass执行期间对submit加锁，使其在该期间不能执行。

这种方案可以真正解决上述依赖问题，对于多依赖的情况则可以设置多个锁变量，因此也不存在问题3的困扰。但这个方案最大的问题就是

  • 变量的增多对其管理的成本增加
  • 代码耦合性急剧增加

如果哪天需求变了：submit依赖的事件增多或减少了，天哪！你确定不喝杯茶冷静冷静再维护？

因此将这些变量封装起来势在必行，经过我多次思考加实践终于出了下面的解决方案：

该方案为每个事件函数增添了三个新属性：

  1. required => 表示该事件所需依赖的事件（为定值，在初始化阶段就确定好）
  2. consumers => 表示该事件执行完毕（包括异步请求）后所需执行的事件 （为变量，动态添加）
  3. state[‘init’, ‘running’, ‘ready’] => 表示该事件当前所处阶段

所有事件函数在初始化时就把其依赖的事件保存在其required中，并标志其state为’init’
所有事件函数执行前都要检测其required中的所有事件是否都处于’ready’状态，是的话就继续执行，否则对于’init’状态的强制执行并把当前的事件添加到其consumers中，对于’running’状态的直接把当前执行的事件添加到其consumers中。
所有事件执行完毕（含有异步请求要等到异步执行结束），检查其consumers中的所有依赖其的事件，尝试执行。

如此一来，便可比较完美解决事件依赖问题，并且维护起来也比较容易（依赖有变动，只需修改ready中initRequire的参数即可）。
但该解决方案还有以下几个问题：

  1. 所有事件函数需要显式检查其required和consumers
  2. 没考虑多层次事件依赖
